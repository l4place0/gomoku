## Context

本项目是一个 15×15 的五子棋博弈引擎，采用 Python Pygame 作为前端 GUI，并通过 ctypes 调用由 C++ 编写的搜索引擎 DLL。
目前引擎支持传统极小极大（MiniMax Alpha-Beta）搜索与 KataGomo 神经网络推理的加权混合。然而，混合机制处于全盘静态阶段，且开局时缺乏开局库支持，导致开局阶段 CPU 算力大量空转；同时，MiniMax 的置换表最佳着法未被有效利用，且缺乏杀手启发与历史启发等先进搜索剪枝技术，限制了搜索深度。

## Goals / Non-Goals

**Goals:**
- 构建一个**三阶段混合博弈流水线**：开局查库 -> 中盘 MCTS 战略大局观 -> 终局 Alpha-Beta + VCF 精准算杀。
- 引入**双源开局库**机制，支持传统职业定式与 AI 在自对弈中探索并沉淀出的高胜率冷门神仙步。
- 采用 **8路对称性归一化** 对开局进行去重和紧凑存储，哈希表检索效率达到 $O(1)$。
- 激活并优化 C++ 端的**置换表 Hash Move**、**双杀手着法（Killer Move）**和**全局历史启发（History Heuristic）**动态排序，提高 Alpha-Beta 搜索深度 2-4 层。
- 设计多维动态框架转换器，在确保战术安全性（无漏招）的前提下释放最高硬件效能。

**Non-Goals:**
- **不**重写神经网络推理层。依然使用现有的 `KataInferenceAdapter` 后端。
- **不**对中盘布局的评估函数进行底层大改。主要依靠神经网络大局观与搜索剪枝效率的提升来拉高棋力。
- **不**在本项目中进行大规模深度神经网络的模型重新训练，只针对推理参数（Visits、Blends）进行动态调优。

## Decisions

### 决策 1：基于 8 路对称性的开局库归一化与哈希检索
- **技术选择**：对于前 8 手内的盘面，使用 8 路对称变换（旋转 90/180/270 度，以及水平、垂直、主副对角线镜像翻转），将第一个偏离天元（7,7）的棋子旋转至第一象限的特定 45 度角（标准姿态）。
- **理由**：五子棋棋盘具有完全的对称性。若不进行归一化，同一个开局定式在 8 个方向上需要重复录入 8 次，库体积膨胀且维护困难。归一化确保了仅需存储一个“标准姿态”的哈希值，大大压缩了体积。
- **备选方案**：存储所有 8 个方向的哈希值。这虽然免去了坐标变换计算，但会导致数据重复，且难以在库中合并不同玩家探索出的同一本质开局。

### 决策 2：C++ 搜索引擎的动态三层走法排序套件
- **技术选择**：在 C++ `negaMax` 递归内部引入动态排序函数 `getMoveSortScore(move, depth, hashMove)`，并在 `genMoves` 静态排序后对候选队列进行重排：
  - **第一层**：如果 `move == hashMove`（置换表中曾记录过的最佳着法），赋予最高分数 `10,000,000`。
  - **第二层**：如果 `move == killerMoves[depth][0]` 或 `killerMoves[depth][1]`，赋予高分 `9,000,000` / `8,000,000`。
  - **第三层**：静态棋型估分 + 历史得分 `historyScore[move.x][move.y]`，合并范围 `[0, 1,000,000]`。
- **理由**：Alpha-Beta 搜索中，如果能优先探索最佳走法或能引发 Beta 剪裁的防御点，可以立即触发剪枝，避开对其他 99% 冗余分支的搜索，这能将相同思考时间下的搜索深度提升 2-4 层。

### 决策 3：动态 Visits 调配与多维时机转换中心
- **技术选择**：在 Python 侧动态控制 visits。
  - 开局（手数量 $\le 6$）完全信任开局库或 KataGomo 高 visits 推理。
  - 中盘（手数量 $7 \sim 15$）采用神经网络 64 visits + Alpha-Beta 6层搜索。
  - 终局（手数量 $> 15$）或者发现对手有活三/冲四时，关停 KataGomo (visits = 0)，将所有资源分配给 12 层以上的 Alpha-Beta 精准算杀与 VCF 校验。
- **理由**：开局太空时搜索无意义，残局激烈时神经网络由于visits分布扁平可能会出现算力盲点。动态 visits 控制和强制切换既避免了开局计算空转，又堵上了终局被战术绝杀的漏洞。

### 决策 4：极轻量级开局库数据格式 (JSON)
- **技术选择**：使用 `opening_book.json` 存储开局库。
- **理由**：在前 8 手内，库的总数据量不超过几万条。使用 JSON 文件可以通过 Python 字典在毫秒级内完成 $O(1)$ 内存查询，且格式极其直观，完美支持跨平台运行，免去了引入复杂 SQLite C++ 库或 Python 第三方数据库依赖的工程开销。

## Risks / Trade-offs

- **[Risk]** AI 自对弈发现的高胜率“奇招”数据存在噪点，可能误将人类测试时的失误定式当做“AI 奇招”存入库中。
  - **Mitigation**：引入**双重过滤阀门**。只有当某个走法的总访问量 `visits > 800`，且胜率 `winrate > 0.65`（或白棋胜率稳定 $>0.53$），且经过 Alpha-Beta 深度验证不为禁手/负分时，才允许导出并沉淀入库。
- **[Risk]** 传统定式在旋转/翻转坐标还原时出现偏差，导致落子位置出错。
  - **Mitigation**：在 Python 侧编写严格的坐标变换矩阵单元测试，覆盖 8 种对称模式的转换和逆变换，确保完全一致。
