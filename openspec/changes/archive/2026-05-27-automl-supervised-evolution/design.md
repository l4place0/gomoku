## Context

当前五子棋博弈引擎的机器学习（ML）管线由一系列相互独立的脚本构成：`selfplay` (由 C++ 编写的 `katago.exe` 执行), `shuffle.py` (数据混洗), `train.py` (PyTorch 训练), `export_model_pytorch.py` (模型导出), 以及 `headless_runner.py` (无头对弈评估)。
这些脚本缺乏统一的编排接口，且海量的详细日志（例如每一局自对弈的走子、训练的逐 Batch 损失）直接喷涌到控制台，使得 Agent 极难提取核心指标进行长期自动化的调参和迭代。另外，模型是否变强完全靠人工点按 UI 来对局，缺乏无人值守的闭环自动 PK 与晋升机制。

## Goals / Non-Goals

**Goals:**
- 提供全新的 `automl_cli.py` 编排程序，能够通过参数完全掌控训练的所有核心阶段（自对弈、混洗、训练、导出、PK、晋升）。
- 在控制台（stdout）中实现极其清爽、高度精炼的步骤进度和汇总日志，将底层原始的繁琐日志默默拦截并分流写入独立的归档日志文件中。
- 强制实施“参数证据链记录”，每次运行均以结构化 Markdown 形式向控制台和日志头部输出所有被拼接的超参数，实现 100% 实验复现。
- 实现基于胜率阈值（如 $\ge 55\%$）的自动化晋升决策（Auto-Promotion）和定式库自进化对接。
- 制作配套的 Agent 长期监督 Skill 配置文件，定义 Agent 的动态参数微调规则和故障自愈逻辑。

**Non-Goals:**
- 重写有监督训练算法本身（即不修改 PyTorch 神经网络核心特征代码）。
- 重构 GUI 对弈页面本身的 UI（这些属于游戏工程侧，不属于自动化 ML 编排侧）。

## Decisions

### 决策 1：离散的“代（Generation）”闭环执行架构
- **方案选择**：相比 KataGo 原生的持续自对弈与持续训练（Continuous RL），我们选择更加轻量、易调试的**离散世代演化**。即每一轮迭代是一个封闭的生命周期：生成新样本 $\to$ 混洗 $\to$ 训练出一代新候选模型 $\to$ 闭环 PK $\to$ 决定晋升或淘汰 $\to$ 开启下一世代。
- **合理性**：更适合个人 GPU/显卡等中低算力设备，能够做到每一步都清晰可控，适合 Agent 进行精细化调参和诊断。

### 决策 2：底细分流与控制台证据链记录
- **方案选择**：当 `automl_cli.py` 被拉起时，使用 `subprocess.Popen` 以管道流式捕获原始子进程输出。所有明细输出（几百万行的 policy/value 评分等）重定向到 `logs/round_{round}_{stage}.log`。控制台只以统一前缀 `[Round N] [Stage] Message` 打印核心进度。
- **合理性**：极大降低 Agent 监听和解析 stdout 的 Token 消耗与计算开销，同时保存完整的实验日志归档，作为无争议的调试凭证。

### 决策 3：PK 阶段的 Policy Temperature 扰动
- **方案选择**：鉴于我们的第一场 headless PK 测试显示：两个确定性模型加上确定性开局会导致每局下出 100% 一模一样的棋谱。我们在 PK 阶段强制引入小幅度的随机落子温度扰动（在 `headless_runner.py` 参数中增加 `--pk-temp` 控制，或在开局阶段使用随机对称的定式变化）。
- **合理性**：确保 20 或 40 局的 PK 对决覆盖不同的盘面路径，使得评估得出的胜率和 ELO 增长具备真正的统计学置信度。

## Risks / Trade-offs

- **[Risk] 显存溢出与 GPU 降频**
  - *Mitigation* → `automl_cli.py` 必须以**严格的串行时序**调度每个阶段（自对弈结束后才启动 PyTorch 训练，训练完且释放 GPU 后才拉起 PK 进程）。Agent 监督 Skill 中若观测到 CUDA Out of Memory 报错，自动拦截并自动降低 batch 大小后重启本轮。
- **[Risk] 随机探索度不足导致自进化陷入死局 (Local Optima)**
  - *Mitigation* → 如果连续 3 轮新模型都无法击败旧模型，Agent 会动态介入，主动增加 `--sf-games` 并提升自对弈 visits，以及适度增加随机温度，强迫模型跳出舒适区探索新棋型。
