## Context

在目前的五子棋项目中，底层的 C++ DLL 引擎（`GameEngine`）采用单例（Singleton）设计，导出的外部接口只服务于单个全局的引擎实例。这就限制了在同一个进程中无法真正并发加载两个完全独立的模型引擎（如加载不同的神经网络权重 `.bin.gz`）。同时，目前 Python 端的 Pygame 循环（`game.py`）使用单线程设计，利用 `GamePhase` 状态机区分人机/人人对弈阶段，任何长耗时的 DLL 搜索调用都会阻塞主 GUI 渲染线程。

为了在无须重构 C++ DLL 的高安全性要求下实现双 AI 对决，本设计方案提出**“动态参数切换与帧级动作延迟时序”**的核心架构。

## Goals / Non-Goals

**Goals:**
- **免底层修改**：无需对 C++ DLL 引擎做任何代码重构或重新编译，保证架构的安全和向后兼容。
- **独立参数个性化**：允许黑白双方 AI 在局中拥有完全独立的 Visits（模拟次数）、Policy/Value 混合配比、或者完全切换至启发式 MiniMax / VCF 搜索。
- **平滑动画与非阻塞体验**：在 AI 回合引入时间戳/帧延迟管理，提供清晰的落子延迟，保证用户能流畅观察每一步棋局的变化，并能在对战期间随时暂停、继续或调整参数。
- **结构化评估可视化**：在侧边栏完整呈现实时对决指标面板，高亮当前正在思考的一方。

**Non-Goals:**
- **真并发多实例支持**：不重构 C++ 引擎以支持同时载入两个不同的 `.bin.gz` 权重文件，这涉及极高的显存开销与多线程 CUDA 冲突崩溃风险。
- **多线程 Ctypes 搜索异步化**：排除使用 Python `threading` 调用底层 C++ 搜索，由于 Python Ctypes 在单例 C++ 引擎中可能存在的重入并发安全问题，采用主线程帧级时间片轮询延迟是更安全的解法。

## Decisions

### 决策一：参数局内瞬时动态注入 vs 引擎实例分离
我们选择**在 Python 回路中，当轮到相应 AI 执子时瞬时通过 Ctypes 动态修改全局引擎参数**。

- **备选方案 (多 DLL 实例)**：重构 C++，取消单例，暴露实例创建与销毁接口。
- **对比与抉择**：KataGomo 的神经网络后端在 Windows 系统下加载两个评估器实例极其不稳定，容易发生显存抢占或 CUDA 初始化冲突。采用“动态注入参数”方案，其切换耗时为 `0ms`（仅需更新 Python 侧的数值并通过 Ctypes 传值），能以 100% 的安全性达到模拟不同棋力 AI 互博的业务目标。

### 决策二：步时延迟机制的设计 —— Pygame 帧计时轮询 vs Python Time Sleep
为了防止两个 AI 在自对弈模式下瞬间下完满盘，我们必须引入延迟。我们选择 **基于 Pygame 的帧时间戳累加轮询（Frame timing loop）** 而非直接调用 `time.sleep()`。

- **对比与抉择**：若使用 `time.sleep()`，会导致整个 Pygame 的 GUI 事件循环被挂起，引发操作系统判定窗口“无响应”（转菊花）且无法进行任何按钮点击（如暂停、调节滑块）。使用帧计时轮询，我们在主循环的每次 `tick` 中判断 `current_time - last_move_time >= delay_ms`，非决策期间依然保持完整的事件侦听，保证随时可以点击暂停或拖拽滑块。

```
          Pygame Main Loop (60 FPS)
     ┌────────────────────────────────┐
     │  处理窗口关闭、鼠标滑块等事件  │
     └───────────────┬────────────────┘
                     ▼
       是否处于 AI_VS_AI 对决模式?
       ├───► 否: 走原有人机/人人流程
       └───► 是:
             Is Paused? ──► 是 ──► 跳过决策，仅渲染
             │ 否
             ▼
             Ticks - LastMoveTicks >= Delay?
             ├───► 否: 跳过决策，仅渲染
             └───► 是:
                   1. 识别当前 Player (Black/White)
                   2. 读取对应 UI 滑块的实时参数
                   3. 调用 SetKataSearchParams 注入 C++
                   4. 调用 GetTopMoves 进行决策落子
                   5. 更新 LastMoveTicks，记录结构化日志
```

## Risks / Trade-offs

- **[Risk]** 高 Visits (如 500+) 决策时主线程仍然会发生局部短暂卡顿（大约 0.5s - 1s）。
  - **Mitigation**: 在 UI 界面限制自对战时的 Visits 最大值为 `512`（可满足普通高水平博弈对比），且在侧边栏显示“思考中...”动态小圆点或发光呼吸灯，向观战用户提供明确的视觉反馈。
- **[Risk]** 在对决进行中玩家疯狂调整 Visits 滑块可能引发的临界抖动。
  - **Mitigation**: 仅在每次准备调用 `GetTopMoves` 之前的那一刹那，从 UI 状态中读取当前滑块的数值并进行底层注入，在搜索执行期间锁定该角色的滑块修改权限。
