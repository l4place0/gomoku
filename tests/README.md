# Gomoku Testing Harness & Verification SOP

本项目建立了完整的 Spec-Driven & Test-Driven (SDD-TDD) 开发闭环保障体系，包含单元测试（Tier 1）、DLL 集成测试（Tier 2）和手动验证指南（Tier 3）。

---

## 🚀 自动验证流水线 (Tier 1 & Tier 2)

所有能够自动验证的代码和接口（纯算法逻辑与 C++ DLL 落子、胜负判定、禁手检测接口）都必须由自动化测试保障。

### 运行环境要求
- Python 3.10+
- 已经编译生成好并置于根目录的 `GameEngine.dll`（Windows）或 `GameEngine.so`（Linux/WSL）
- 已安装 `uv` 依赖管理器

### 一键执行指令
在项目根目录下，使用 `uv` 运行全部测试用例：
```bash
uv run pytest
```

---

## ✍️ 手动验证 SOP 指南 (Tier 3)

对于由于显示设备（Pygame GUI）、GPU 加速（CUDA）或长时间神经网络博弈而无法在 headless CI 环境下进行完全自动化断言的功能，开发者或 AI Agent **必须**按照以下 SOP 指南进行手动确认，并在变更日志中记录确认结果。

### 1. Pygame GUI 交互与渲染验证

#### [测试场景：启动与交互界面渲染]
- **WHEN**: 在有物理显示设备的系统上运行 `uv run python game.py`
- **THEN**: 
  1. 应该能够看到木纹质感的 15×15 棋盘正常渲染，四周没有黑边或拉伸。
  2. 左上角显示“五子棋 AI 对弈系统”标题。
  3. 鼠标划过棋盘时，应显示半透明落子提示预览。

#### [测试场景：悔棋(Undo)功能验证]
- **WHEN**: 在棋盘上下几步子，然后点击右下角的“悔棋”按钮
- **THEN**: 
  1. 棋盘上最后一个落子被成功移除，并且 DLL 侧状态和 Python 侧状态均回到上一回合状态。
  2. 控制台或运行日志 `logs/runtime.log` 应记录 `UndoMove` 调用日志。

---

### 2. CUDA 模式与神经网络推理验证

#### [测试场景：KataGomo 权重与 CUDA 推理运行]
- **WHEN**: 确保 `model_weights.txt` 存在并且启动系统
- **THEN**: 
  1. 控制台或 `logs/runtime.log` 应正确打印加载日志：`[KataGomo] 已启用`。
  2. 与电脑对弈时，电脑思考时间应与 `visits` 设置相符，且没有出现 CUDA 显存溢出或程序假死。
  3. 思考过程中，控制台输出神经网络的各落子点 policy 与 value 估值评估日志。

---

### 3. 双 AI 自对弈模式验证

#### [测试场景：自对弈开始与暂停控制]
- **WHEN**: 在游戏主页面选择“AI 对弈模式”，选择黑白双方参数，并点击“开始自对弈”
- **THEN**:
  1. 棋盘上黑白棋子开始自动轮流落子，且每次落子之间有明显的延迟等待时间（默认为 500ms）。
  2. 点击“暂停/继续”按钮，自对弈能够平滑地挂起与继续，不发生多线程竞争性死锁崩溃。
